    Benifits:

    • When editing online bash shell command; its keystrokes are inspired by emacs.
    • AI will always be an important branch of computer application. The basic emacs editing functionalities are implemented by C primitives. Elisp, (Emacs Lisp), language is designed to glue these functionalities, and provides the intelligence of emacs editor. 
      $ ls -l /usr/local/share/emacs/21.4/lisp/*.el
    • Guile, a dialect of scheme language, (GNU Extension Language), is integrated in our emacs-21.4. Via run-ilisp command in our emacs, we can run guile inside an Emacs buffer. (Of course, you need to load the ilisp library before running this command.)
    • We extend guile functionality, (via source in dsc subdirectory, written in C++), to include the capability of sgml document parsing. The grove (Graph Representation of Property Values) data model is an ISO standard to represent a parsed SGML document in memory. This is one way to produce Graph Stores NoSql database. The other three types of NoSql Database are Key-Value stores, Column stores, and Document Stores Databases.

  2. Our BigData Cloud

    Prologue: Two years ago, Professor Yen acquired a Raspberry PI2 SBC for me. Since then, from time to time, I have been trying to adopt our Virtualization knowhow to Rpi2. Basically, our accomplishment is: We can create VMs in x86_64 to cross compile software for arm-based SBCs. This was a big achievement, since software compilation in Rpi2 is painfully slow and errorprone business. Coping the generated binary to uSD, booting the arm SBC, testing the binary is really a nightmare. Via the qemu-system-arm command, we can first test our compiled binary in our x86_64 host before duplicating it to uSD and booting up the SBC.

    By the way, cross compilation in x86_64 is much faster and the software development environment is much richer, i.e. better equipped. In the last new year holiday, (2017), after the last time I update the software for Rpi2, the whole machine is dead completely. The only way to fix it is to downgrade software packages or reinstall the system. This is the first time, the Debian Organization fails on me, the used to be Univarsal Operating System Organization.

    • Hardware ($59 only, Gee!)
      • 8 cores,

      • 2GB ram,

      • With 1 (or 2) TB external USB 3.0 disk,

      2GB ram: Each machine equips with 1T USB3 disk. For each 1G storage space, we need 1M memory to cache its metadata: Such as: The address of this space. Is it free? Who uses it? If it is a file, the file attribute, etc. 1G memory is reserved for ceph. We have only 1G free memory left. For optimal performance, each machine can contribute 2 to 4 VMs for data analytics or database management purpose.

      Do we need 128GB USB 3.0 flash disk? VM templates do need a lot of space. If we do get this flash disk, our VMs will be placed in here, but no more USB 3.0 port! Also, do we have enough power supply for USB 3.0 TB disk and USB 3.0 flash disk simultaneously?
